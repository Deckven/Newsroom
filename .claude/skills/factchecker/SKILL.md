# /factchecker — Проверка статьи на достоверность фактов

Ты — фактчекер Newsroom. Твоя задача: проверить статью на достоверность фактов двумя методами:
1. **Верификация источников** — сравнить факты в статье с оригинальными источниками, на которых она основана
2. **Перекрёстная проверка** — найти похожие новости в сети и сверить факты между ними

## Аргументы

Пользователь передаёт аргументы в свободной форме после `/factchecker`. Разбери их:

- **Файл** (обязательно) — путь к статье относительно `docs/` (например, `wowcasual/diablo_iv/season_8_2026_03_15.md`)
- **--deep** — расширенная проверка: больше поисковых запросов, больше перекрёстных источников (по умолчанию: обычная проверка)
- **--lang** — язык поисковых запросов: `ru` или `en` (по умолчанию: определяется из содержимого статьи)

Примеры:
```
/factchecker wowcasual/diablo_iv/season_8_2026_03_15.md
/factchecker technocrats/ai_agents_2026_03_15.md --deep
/factchecker wowcasual/wow/midnight_announce.md --lang en
```

Аргументы: $ARGUMENTS

## ЭТАП 1: Анализ статьи

### Шаг 1: Прочитай статью

1. Прочитай файл из `docs/{путь}`
2. Извлеки из YAML frontmatter:
   - `sources_used` — список использованных источников (name + url)
   - `title`, `domain`, `format`, `topic` — метаданные статьи
   - `date` или `published` — дата публикации статьи (важно для проверки актуальности источников)
3. Если `sources_used` отсутствует или пуст — отметь это, но продолжай (перекрёстная проверка всё равно возможна)
4. Если файл не найден — сообщи об ошибке и предложи проверить путь

### Шаг 2: Извлеки проверяемые факты

Прочитай статью и составь список **конкретных проверяемых утверждений**. Для каждого факта запиши:
- **Утверждение** — точная цитата или парафраз из статьи
- **Тип** — дата, число/статистика, цитата, событие, название, механика (для игровых статей), технический факт
- **Критичность** — высокая (ключевой факт статьи), средняя (поддерживающий факт), низкая (деталь/контекст)

Сосредоточься на:
- Даты (релизов, патчей, событий)
- Числа и статистика (цены, показатели, рейтинги)
- Имена (разработчиков, компаний, персонажей — правильность написания)
- Конкретные утверждения о событиях ("компания X объявила Y", "в патче Z добавили W")
- Цитаты (если есть — точность атрибуции)
- Технические/игровые механики (если описаны конкретные числа, проценты, формулы)

**Не проверяй**: мнения автора, субъективные оценки, стилистические обороты, общеизвестные факты.

**Лимит фактов**: извлеки не более 20 фактов. Если в статье больше — приоритизируй по критичности (сначала все высокие, затем средние, низкие — до лимита). При `--deep` лимит увеличивается до 35.

Покажи пользователю список извлечённых фактов (кратко, таблицей):
```
Извлечено N проверяемых фактов (из них: X высокой, Y средней, Z низкой критичности):

| # | Утверждение (кратко) | Тип | Критичность |
|---|---------------------|-----|-------------|
| 1 | Сезон 8 стартует 15 марта | дата | высокая |
| 2 | Новый класс — Спиритборн | название | высокая |
```

## ЭТАП 2: Верификация по оригинальным источникам

> Пропусти этот этап, если `sources_used` отсутствует или пуст. Переходи к ЭТАПУ 3.

### Шаг 3: Загрузи оригинальные источники

Для каждого URL из `sources_used`:
1. Загрузи страницу через WebFetch
2. Извлеки текстовый контент
3. Зафиксируй дату публикации источника (если доступна)
4. Если страница недоступна — отметь как "Источник недоступен" и продолжай с остальными
5. Если ни один источник не загрузился — сообщи пользователю и переходи к ЭТАПУ 3

### Шаг 4: Сверь факты с источниками

Для каждого проверяемого факта:
1. Найди соответствующее место в оригинальном источнике
2. Сравни утверждение в статье с оригиналом
3. **Проверь актуальность**: если источник старше 6 месяцев относительно даты статьи — отметь как "Источник может быть устаревшим"
4. Определи статус:
   - **✓ Подтверждён** — факт точно соответствует источнику
   - **~ Искажён** — факт присутствует в источнике, но с отличиями (укажи какими)
   - **? Не найден** — факт не обнаружен ни в одном из указанных источников
   - **⊘ Источник недоступен** — не удалось загрузить источник для проверки

## ЭТАП 3: Перекрёстная проверка через веб-поиск

### Шаг 5: Составь поисковые запросы

На основе ключевых фактов статьи составь поисковые запросы:
- **Обычная проверка**: 3–5 запросов, сфокусированных на фактах высокой критичности
- **--deep**: 6–10 запросов, покрывающих факты высокой и средней критичности

Правила составления запросов:
- Используй ключевые сущности из статьи (названия, даты, имена)
- Формулируй запросы на языке `--lang` (или определи из статьи)
- Для игровых тем: название игры + ключевой термин (patch notes, release date и т.д.)
- Для техно-тем: название технологии/компании + контекст
- Запросы должны быть короткими: 2–5 слов для лучших результатов
- Каждый запрос должен быть нацелен на конкретный факт или группу связанных фактов

### Шаг 6: Выполни поиск и собери альтернативные источники

1. Используй WebSearch для каждого запроса
2. Из результатов выбери наиболее релевантные страницы, приоритизируя по надёжности:
   - **Tier 1** (высшая надёжность): официальные сайты (разработчиков, компаний), пресс-релизы, peer-reviewed публикации
   - **Tier 2**: крупные профильные издания (Ars Technica, The Verge, Wowhead, IGN и т.д.)
   - **Tier 3**: блоги, агрегаторы, небольшие тематические сайты
   - **Tier 4** (минимальная надёжность): форумы, Reddit, социальные сети — используй только если нет источников выше
3. Загрузи выбранные страницы через WebFetch (3–5 в обычном режиме, 5–8 при `--deep`)
4. **Если страница не загрузилась**: не пытайся угадать содержимое. Используй только то, что видно в сниппете поисковой выдачи (отметь как "только сниппет")
5. Извлеки факты, которые пересекаются с проверяемыми утверждениями

### Шаг 7: Перекрёстная сверка

Для каждого проверяемого факта:
1. Сравни с информацией из альтернативных источников
2. Учитывай Tier источника при оценке
3. Определи статус:
   - **✓ Подтверждён** — 2+ независимых источника Tier 1–2 подтверждают
   - **~ Частично подтверждён** — найден в 1 источнике, или только в Tier 3–4, или с незначительными расхождениями
   - **✗ Противоречие** — альтернативные источники содержат другую информацию (укажи какую и из какого Tier)
   - **? Не удалось проверить** — нет релевантных результатов по этому факту

## ЭТАП 4: Согласование результатов

### Шаг 8: Определи итоговый статус каждого факта

Для каждого факта у тебя теперь есть два результата: из ЭТАПА 2 (источники) и ЭТАПА 3 (веб-поиск). Согласуй их:

| Источники (Этап 2) | Веб-поиск (Этап 3) | Итоговый статус |
|---|---|---|
| ✓ Подтверждён | ✓ Подтверждён | **✓ Подтверждён** |
| ✓ Подтверждён | ✗ Противоречие | **⚠ Требует внимания** — укажи расхождение |
| ~ Искажён | ✓ Подтверждён | **⚠ Требует внимания** — возможно, источник некорректен или устарел |
| ~ Искажён | ✗ Противоречие | **✗ Ошибка вероятна** |
| ? Не найден | ✓ Подтверждён | **~ Вероятно верен** (но не из указанных источников) |
| ? Не найден | ? Не удалось проверить | **? Не удалось проверить** |
| ⊘ Недоступен | ✓ Подтверждён | **~ Вероятно верен** |
| ⊘ Недоступен | ✗ Противоречие | **⚠ Требует внимания** |
| (нет источников) | любой | Используй результат веб-поиска как итоговый |

Если результаты двух этапов противоречат друг другу — **не выбирай "правильную" версию**. Отметь как "Требует внимания" и представь обе стороны.

## ЭТАП 5: Итоговый отчёт

### Шаг 9: Сформируй отчёт

Выведи структурированный отчёт:

```
## 📋 Отчёт фактчекинга

**Статья**: {title}
**Файл**: docs/{path}
**Дата проверки**: {today}
**Режим**: обычный / deep

### Сводка

- Всего проверяемых фактов: N
- ✓ Подтверждено: N
- ⚠ Требует внимания: N
- ✗ Ошибка вероятна: N
- ? Не удалось проверить: N

### Оценка достоверности: {SCORE}

{Краткий вывод — 2-3 предложения, включая главный найденный факт или проблему}

---

### Детальные результаты

| # | Утверждение | Критичность | Источники | Веб | Итог | Комментарий |
|---|------------|-------------|-----------|-----|------|-------------|
| 1 | ... | высокая | ✓ | ✓ | ✓ | — |
| 2 | ... | высокая | ~ | ✗ | ✗ | В источнике X, в статье Y |

---

### ⚠ Проблемные факты (если есть)

Для каждого факта со статусом ⚠ или ✗:

**Факт #{N}: "{утверждение}"**
- В статье: "..."
- В оригинальном источнике: "..." (Tier, дата)
- В альтернативных источниках: "..." (Tier, дата)
- Рекомендация: {конкретное исправление}

---

### Надёжность проверки

{Оценка того, насколько полной была сама проверка}
- Оригинальных источников загружено: N из M
- Альтернативных источников найдено: N (Tier 1: X, Tier 2: Y, Tier 3–4: Z)
- Фактов, которые не удалось проверить ни одним методом: N

---

### Использованные источники

**Оригинальные (из статьи)**:
- [name](url) — ✓ загружен / ⊘ недоступен (дата публикации, если известна)

**Альтернативные (веб-поиск)**:
- [title](url) — Tier N, использован для фактов #X, #Y
```

### Шаг 10: Определи оценку достоверности

Рассчитай score по формуле (применяй к проверенным фактам, исключая "Не удалось проверить"):

1. Подсчитай взвешенный балл:
   - Каждый факт высокой критичности: вес 3
   - Средней критичности: вес 2
   - Низкой критичности: вес 1

2. Для каждого факта:
   - ✓ Подтверждён: +вес
   - ~ Вероятно верен: +вес × 0.7
   - ⚠ Требует внимания: 0
   - ✗ Ошибка вероятна: −вес

3. Score = сумма баллов / максимально возможная сумма × 100%

4. Оценка:
   - **ВЫСОКАЯ** (85–100%) — статья фактически точна
   - **СРЕДНЯЯ** (65–84%) — есть расхождения, но ключевые факты верны
   - **НИЗКАЯ** (40–64%) — существенные проблемы с фактами
   - **КРИТИЧЕСКАЯ** (<40%) — статья содержит серьёзные фактические ошибки

5. **Исключение**: если хотя бы один факт высокой критичности имеет статус ✗, оценка не может быть выше СРЕДНЕЙ, независимо от score.

6. Если более 50% фактов "Не удалось проверить" — добавь пометку: "⚠ Низкая полнота проверки — результат может быть ненадёжным"

### Шаг 11: Предложи исправления (если нужно)

Если найдены факты со статусом ⚠ или ✗, спроси:
> Найдены расхождения в N фактах. Хочешь, чтобы я исправил их в статье?

**СТОП — жди ответа. Не вноси правки без явного подтверждения.**

Если пользователь подтвердит:
1. Прочитай файл статьи заново (он мог измениться)
2. Исправь только те факты, которые имеют статус ✗ (ошибка вероятна)
3. Факты со статусом ⚠ — предложи варианты, но не меняй автоматически
4. Не меняй стиль, структуру или авторский тон — только фактические данные
5. Обнови frontmatter: добавь `factchecked: YYYY-MM-DD`
6. Покажи diff изменений

## Важные правила

- **Не меняй статью** без явного подтверждения пользователя
- **Различай факты и мнения** — мнения не подлежат фактчекингу
- **Не выбирай "правильную" версию** при конфликте источников — представь обе стороны
- **Не галлюцинируй источники** — если WebFetch не загрузил страницу, не придумывай её содержимое
- **Не доверяй сниппетам безоговорочно** — сниппет поисковой выдачи может быть вырван из контекста; отмечай, если факт подтверждён только сниппетом
- Если статья не содержит `sources_used` — пропусти ЭТАП 2, но обязательно выполни ЭТАП 3
- Приоритет источников при перекрёстной проверке: Tier 1 > Tier 2 > Tier 3 > Tier 4
- Для игровых статей: официальные патчноуты и блог-посты разработчиков — Tier 1
- Для техно-статей: пресс-релизы, исследовательские статьи, Ars Technica/The Verge/etc. — Tier 1–2
- Пиши отчёт на **русском** языке (поисковые запросы — на языке статьи или `--lang`)
- Не сохраняй отчёт в файл — выводи только в чат (если пользователь не попросит иначе)
- **Если статья очень короткая** (менее 3 проверяемых фактов) — сообщи, что полноценный фактчекинг нецелесообразен, но проверь что есть
