## 1. Обзор и бизнес-контекст

Финтех-компании ежедневно обрабатывают огромные объёмы финансовых документов: счета-фактуры, банковские выписки, ID-карты клиентов, договоры, регуляторные отчёты, результаты KYC/AML-проверок. Традиционная обработка этих документов — ручная или полуавтоматическая — приводит к:

- **Высоким операционным расходам** — ручной ввод данных, проверка и маршрутизация занимают часы
- **Ошибкам и задержкам** — человеческий фактор при работе с неструктурированными документами
- **Комплаенс-рискам** — невозможность масштабировать проверки при росте клиентской базы
- **Информационной разобщённости** — данные из документов «застревают» в отдельных системах

Gemini Enterprise в связке с Google Cloud предлагает автоматизированный конвейер, который решает каждую из этих проблем через три ключевых этапа: **распознавание → категоризация → анализ**.

### Масштаб возможностей

| Параметр | Характеристика |
|:---|:---|
| **Контекстное окно** | До 1 000 000 токенов (Gemini 2.5 Pro) — анализ крупных документов целиком |
| **Модели** | Gemini 3.1 Pro, Gemini 3 Flash, Gemini 3.1 Flash-Lite |
| **Хранение и индексация** | До 75 ГБ на лицензию (Gemini Enterprise Standard/Plus, объединяется в пул) |
| **API-запросы** | 2000 запросов/день на пользователя (Code Assist Enterprise), 120/мин |
| **Загрузка файлов** | До 2 ГБ на файл через Gemini File API, квота 20 ГБ на проект |
| **Резидентность данных** | US, EU или глобальное размещение |

## 3. OCR нового поколения

### Gemini vs традиционный OCR

В отличие от традиционных систем распознавания (Tesseract, ABBYY, Adobe Acrobat OCR), которые работают на уровне символов и шаблонов, Gemini понимает **контекст и семантическую структуру документа**. Это принципиальное отличие:

| Критерий | Традиционный OCR | Gemini OCR |
|:---|:---|:---|
| **Распознавание текста** | Символ за символом, зависит от качества скана | Семантическое понимание контекста |
| **Неструктурированные документы** | Требует шаблонов и предварительной разметки | Работает «из коробки» с любой структурой |
| **Таблицы и формы** | Часто ломает структуру | Сохраняет связи между полями |
| **Рукописный текст** | Низкая точность | Высокая точность благодаря мультимодальности |
| **Мультиязычность** | Требует отдельных моделей | 100+ языков в одной модели |
| **Извлечение смысла** | Только текст | Текст + семантика + связи между данными |

### Как работает Gemini OCR для финансовых документов

1. **Загрузка документа** — файл помещается в Cloud Storage bucket или передаётся через Gemini File API
2. **Мультимодальный анализ** — Gemini обрабатывает документ как визуальный объект, понимая:
   - Заголовки и разделы
   - Таблицы с числовыми данными
   - Подписи и печати
   - QR-коды и штрих-коды
   - Логотипы и реквизиты
3. **Структурированный вывод** — результат возвращается в формате JSON с чётким маппингом полей

## 4. Интеллектуальная категоризация и маршрутизация

### Автоматическое сопоставление с учётными кодами

После извлечения данных из документа Gemini выполняет интеллектуальную категоризацию:

- **Бухгалтерские коды** — автоматическое сопоставление позиций счетов с планом счетов организации
- **Категории расходов** — классификация по центрам затрат (CAPEX/OPEX, отдел, проект)
- **Налоговые категории** — определение ставки НДС, применимых вычетов
- **Регуляторные метки** — тегирование для compliance (PCI DSS, GDPR, DORA)

### Маршрутизация документов

На основе категоризации документы автоматически направляются в соответствующие потоки:

| Тип документа | Маршрут | Действие |
|:---|:---|:---|
| Счёт-фактура < $10,000 | Бухгалтерия → автосогласование | Запись в BigQuery |
| Счёт-фактура ≥ $10,000 | CFO → ручное согласование | Задача в Jira |
| ID-карта клиента | KYC-конвейер | Верификация + профиль |
| Банковская выписка | Reconciliation | Сверка с внутренними данными |
| Регуляторный запрос | Compliance-офицер | Legal Hold в Vault |
| Подозрительная транзакция | Fraud detection | Алерт + блокировка |

### Реализация через Agent Designer

Для бизнес-пользователей без навыков программирования Agent Designer позволяет создать агента категоризации:

1. **Создание агента** — в интерфейсе Gemini Enterprise → Agent Designer
2. **Инструкции**: «Ты — агент категоризации финансовых документов. Получив документ, определи его тип, извлеки ключевые данные и направь в соответствующий поток обработки»
3. **Подключение источников** — Google Drive (входящие), BigQuery (справочник кодов), Jira (задачи)
4. **Расписание** — запуск каждые 15 минут для обработки новых документов

## 5. Анализ аномалий и fraud detection

### Интеллектуальный анализ транзакций

Gemini сравнивает данные из новых документов с историческими паттернами, мгновенно помечая подозрительные транзакции для проверки отделом безопасности.

**Три уровня анализа:**

1. **Статистические аномалии** — суммы, частота, география, контрагенты, отклоняющиеся от исторических паттернов
2. **Семантические аномалии** — несоответствия в описаниях, подозрительные формулировки в договорах
3. **Кросс-документные аномалии** — расхождения между счетами, выписками и платёжными поручениями

### Результаты в индустрии

- **Macquarie Bank** — 40% сокращение ложноположительных срабатываний в fraud detection
- **38% больше** пользователей перенаправлено на self-service (Macquarie)
- **WEX** (платёжные решения) — 63,000 часов экономии ежемесячно

### Реализация через ADK: мультиагентная система fraud detection

Для сложных сценариев обнаружения мошенничества используется Agent Development Kit с мультиагентной архитектурой:

```python
from google.adk.agents import LlmAgent
from adk.mcp.toolset import McpToolset

# Подключение к транзакционной БД через MCP Toolbox
mcp_tools = McpToolset(mcp_server_url="http://localhost:8080")

# Агент анализа транзакций
transaction_analyzer = LlmAgent(
    model="gemini-3.1-pro",
    instruction="""Ты — агент для анализа транзакций.
    Используй доступные инструменты для выявления аномалий:
    - Сравнивай суммы с историческими средними
    - Проверяй частоту транзакций контрагента
    - Оценивай географию операций""",
    tools=mcp_tools.get_all_tools()
)

# Агент проверки контрагентов
counterparty_checker = LlmAgent(
    model="gemini-3-flash",
    instruction="""Ты — агент проверки контрагентов.
    Проверяй контрагентов по санкционным спискам,
    реестрам юрлиц и внутренним чёрным спискам.""",
    tools=mcp_tools.get_all_tools()
)

# Оркестратор
fraud_orchestrator = LlmAgent(
    model="gemini-3.1-pro",
    instruction="""Ты — главный агент обнаружения мошенничества.
    Координируй работу подчинённых агентов:
    1. Направь транзакцию на анализ аномалий
    2. При обнаружении аномалии — проверь контрагента
    3. Сформируй итоговый отчёт с уровнем риска""",
    sub_agents=[transaction_analyzer, counterparty_checker]
)
```

**Конфигурация tools.yaml для MCP Toolbox:**

```yaml
sources:
  - name: transactions_db
    type: postgresql
    uri: postgresql://user:password@host:port/fintech_db

toolsets:
  - name: transaction_analysis_tools
    source: transactions_db
    tools:
      - name: get_transactions_by_user
        description: "Получить транзакции для указанного пользователя"
        query: "SELECT * FROM transactions WHERE user_id = :user_id
                ORDER BY transaction_date DESC LIMIT :limit;"
      - name: get_anomalous_transactions
        description: "Найти транзакции, отклоняющиеся от среднего"
        query: "SELECT * FROM transactions
                WHERE amount > (SELECT AVG(amount) * 3 FROM transactions
                WHERE user_id = :user_id)
                AND user_id = :user_id;"
      - name: check_counterparty
        description: "Проверить контрагента по санкционному списку"
        query: "SELECT * FROM sanctions_list
                WHERE name ILIKE :counterparty_name;"
```

## 6. Финансовая аналитика в Google Sheets

Gemini в Google Sheets позволяет аналитикам переходить от ручного ввода формул к запросам на естественном языке. Для расчёта финансовых показателей достаточно описать задачу — модель выполнит вычисления и объяснит результаты.

### Примеры промптов для финансового аналитика

#### 1. Анализ транзакционных логов

**Контекст:** Данные загружены в Google Sheet. Колонки: `timestamp`, `user_id`, `event_type`, `item_id`, `price`, `device_type`.

**Промпт:**
> Проанализируй данные в диапазоне A1:F50000. Построй воронку конверсии: какой процент пользователей, просмотревших товар, добавил его в корзину, и какой процент из добавивших совершил покупку? Определи топ-5 самых просматриваемых item_id, которые имеют самый низкий процент добавления в корзину.

#### 2. Поиск аномалий в транзакциях

**Промпт:**
> Выступи в роли аналитика данных. Проанализируй транзакции за последний месяц (диапазон A2:E5000). Найди 5 самых значимых аномалий или выбросов в данных по сумме транзакции. Для каждой аномалии укажи user_id и transaction_date.

#### 3. A/B-тестирование

**Контекст:** Колонки: `transaction_id`, `amount`, `user_segment`, `ab_test_variant` ('A' или 'B'), `timestamp`.

**Промпты:**
> 1. Рассчитай среднюю сумму транзакции для вариантов A и B. Есть ли статистически значимая разница?
> 2. На основе результатов предложи гипотезу для следующего A/B теста по увеличению среднего чека.
> 3. Создай сводную таблицу: общая сумма транзакций по каждому user_segment для вариантов A и B.

#### 4. Расчёт финансовых KPI

**Промпт:**
> Проанализируй загруженный балансовый отчёт. Рассчитай: коэффициент текущей ликвидности, коэффициент быстрой ликвидности, рентабельность собственного капитала (ROE), отношение долга к капиталу. Объясни причины изменения показателей по сравнению с предыдущим периодом.

### Ограничения

- Sheets AI эффективен для **десятков тысяч строк**
- Для миллионов записей — предварительная агрегация в **BigQuery**, затем импорт результатов в Sheets
- Gemini лучше работает с «чистыми» данными: чёткие заголовки, без объединённых ячеек
- Для очень больших датасетов рекомендуется работать по частям (порциями по 10,000 строк)

## 7. Комплаенс-автоматизация и Google Vault

### Google Vault для финансовых организаций

Для финтех-компаний, подчиняющихся регуляторам FINRA, SEC (правило 17a-4), DORA, EBA, критически важно обеспечить хранение и экспорт всех коммуникаций, включая AI-сгенерированный контент.

**Google Vault** — инструмент для управления информацией и eDiscovery в Google Workspace:

| Функция | Описание | Значимость для финтеха |
|:---|:---|:---|
| **Retention Rules** | Правила хранения данных (по умолчанию и кастомные) | SEC 17a-4: хранение 7+ лет |
| **Legal Holds** | Блокировка данных при расследованиях | Приоритет над правилами хранения |
| **eDiscovery** | Поиск по ключевым словам, пользователям, датам | Ответ на запросы регуляторов |
| **WORM** | Write Once, Read Many — неизменяемое хранение | Требование FINRA/SEC |
| **Export** | Экспорт в стандартных форматах (MBOX, CSV, XML) | Предоставление данных аудиторам |

### Что покрывает Vault в контексте Gemini

- **Расшифровки и сводки Google Meet** — документы «Сделать заметки» сохраняются в Drive → подпадают под Vault
- **Сообщения в Google Chat** — включая сгенерированные Gemini сводки
- **Промпты и ответы в приложении Gemini** — Vault поддерживает поиск и экспорт
- **Документы, созданные с помощью Gemini** — Docs, Sheets, Slides

### Пошаговая настройка для FINRA / SEC 17a-4

1. **Вход в Google Vault** → vault.google.com
2. **Настройка правил хранения:**
   - Раздел «Хранение» (Retention)
   - Google Drive — хранить 7 лет (2555 дней) или бессрочно
   - Google Chat — аналогичный срок
   - Кастомные правила для OU «Трейдеры» — более строгая политика
3. **Настройка экспорта FINRA:**
   - Требуется Enterprise Plus + Assured Controls
   - Тип экспорта: FINRA
   - Данные → Google Cloud Storage bucket → AODocs (неизменяемые GCS-бакеты)
4. **Настройка Legal Holds:**
   - Создание «дела» (matter) при запросе регулятора
   - Наложение Legal Hold на данные конкретных пользователей
   - Hold имеет приоритет над retention rules — данные не удаляются до снятия Hold
5. **eDiscovery:**
   - Поиск по ключевым словам, пользователям, диапазону дат
   - Экспорт результатов с метаданными
   - Предоставление регуляторам

## 8. Интеграция с корпоративными системами

### Permissions-aware коннекторы

Gemini Enterprise интегрируется с корпоративными системами через коннекторы, которые **учитывают права доступа**. Это критически важно при обработке финансовых документов.

**Механизм работы:**

1. **Извлечение (Fetch)** — сбор контента и метаданных из внешней системы
2. **Преобразование (Transform)** — конвертация в стандартизированный формат с ACL
3. **Синхронизация (Sync)** — загрузка и регулярное обновление в Data Store

**Два режима синхронизации:**
- **Федеративный поиск** — без копирования данных в Google (для строгих требований к локализации)
- **Индексация** — копирование для быстрого и глубокого анализа

### Поддерживаемые системы

| Тип | Системы | Механизм | Применение |
|:---|:---|:---|:---|
| **Productivity** | Google Workspace, Microsoft 365 | API | Поиск по письмам, встречам, документам |
| **Development** | Jira, Confluence, GitHub, GitLab | OAuth 2.0, Webhooks | Связь задач с кодом и документацией |
| **Business Ops** | Salesforce, SAP, ServiceNow | Data Ingestion | Анализ клиентских путей, операционные риски |
| **Data Lakes** | BigQuery, Vertex AI Search | Federated Search | Прямые запросы к транзакционным БД |
| **On-Premise** | Локальные Jira, GitLab | App-MCP, Hybrid NEG | Безопасный мост к внутренним системам |

### Безопасное подключение к PostgreSQL

Для финтех-компаний с транзакционными БД — Enterprise-подход:

1. **Сетевая изоляция** — Cloud SQL for PostgreSQL и агент в одной VPC
2. **Service Account** — выделенный аккаунт с IAM-ролью `Cloud SQL Client` (без паролей)
3. **Cloud SQL Auth Proxy** — шифрованный туннель, агент подключается как к localhost
4. **Secret Manager** — строки подключения и секреты не хардкодятся
5. **MCP Toolbox for Databases** — параметризованные SQL-шаблоны, LLM не имеет прямого доступа к БД

## 9. Практические сценарии для финтеха

### Сценарий 1: Автоматизация KYC-проверки (Agent Designer)

**Инструменты:** Agent Designer (no-code)

1. Клиент загружает ID-карту и подтверждающие документы через портал
2. Документы поступают в Google Drive → триггер для агента
3. **Агент KYC-проверки:**
   - Извлекает данные из ID (ФИО, дата рождения, номер документа)
   - Сверяет с санкционными списками
   - Проверяет соответствие данных в заявке
   - Генерирует отчёт о проверке
4. При положительном результате — автоматическая регистрация через AppSheet
5. При обнаружении расхождений — задача в Jira для ручной проверки

### Сценарий 2: Обработка входящих счетов-фактур

**Инструменты:** n8n + Gemini API

1. Поставщик отправляет счёт на email → Gmail
2. n8n workflow извлекает вложение → Cloud Storage
3. Gemini OCR → структурированные данные (JSON)
4. Валидация: проверка ИНН контрагента, кросс-проверка сумм
5. Категоризация: автоматическое присвоение бухгалтерского кода
6. Маршрутизация: < $10K → автосогласование; ≥ $10K → согласование CFO
7. Запись в BigQuery для аналитики и аудита

### Сценарий 3: Генерация compliance-отчёта по коду

**Инструменты:** Gemini Enterprise + SonarQube + Confluence

1. Разработчик отправляет код в GitLab → CI/CD pipeline
2. SonarQube выполняет SAST-сканирование
3. Gemini-агент получает:
   - Код из merge request (GitLab)
   - Отчёт об уязвимостях (SonarQube)
   - Стандарты безопасности (Confluence)
4. Генерирует структурированный отчёт:

**Пример вывода:**

> **Отчёт о комплаенс-проверке Merge Request #123**
>
> **1. Критическое замечание (нарушение PCI DSS):**
> - Файл: `PaymentService.java`, строка 45
> - Проблема: логирование полного номера банковской карты
> - Рекомендация: маскирование — оставить видимыми первые 6 и последние 4 цифры
>
> **2. Средний приоритет (отступление от стандарта кодирования):**
> - Файл: `UserController.java`, метод `getUserData`
> - Проблема: отсутствует обработка `NullPointerException`
> - Рекомендация: добавить проверку на null перед вызовом `getProfile()`
>
> **Общий вывод:** Merge request не может быть одобрен до устранения критического замечания.

5. Отчёт прикрепляется к MR в GitLab, задача в Jira автору

### Сценарий 4: Мониторинг регуляторных изменений

**Инструменты:** Deep Research + NotebookLM

1. **Deep Research** — автоматическое отслеживание изменений в регуляторике
2. **NotebookLM** — загрузка нормативных документов (GDPR, EBA, DORA):
   - Синтез знаний: мгновенные ответы о соответствии новых функций нормам
   - Анализ Earnings Calls: выделение трендов и угроз из транскриптов конкурентов
   - Источник правды: ссылки на конкретные страницы и абзацы — исключает галлюцинации
3. Автоматическая генерация compliance-отчётов для регуляторов

## 11. Безопасность обработки финансовых данных

### Ключевые гарантии

- **Данные клиентов НЕ используются для обучения моделей** — промпты и контент остаются собственностью клиента
- **Client-side encryption** — шифрование на стороне клиента
- **CMEK** — клиент управляет ключами шифрования (требует мультирегион US или EU)
- **VPC Service Controls** — предотвращение утечки данных
- **Model Armor** — защита от adversarial атак на модели
- **DLP policies** — автоматическое обнаружение и блокировка чувствительных данных

### Архитектура безопасности для обработки документов

```
┌────────────────────────────────────────────────┐
│           ПЕРИМЕТР БЕЗОПАСНОСТИ                │
│  VPC Service Controls + CMEK + DLP             │
├────────────────────────────────────────────────┤
│  Аутентификация: SSO (SAML 2.0 / OIDC)       │
│  Авторизация: RBAC + permissions-aware search  │
│  Шифрование: at rest + in transit + CSE        │
├────────────────────────────────────────────────┤
│  Аудит: Vault + журналы доступа               │
│  Retention: WORM + Legal Holds                 │
│  Export: FINRA/SEC формат через AODocs          │
└────────────────────────────────────────────────┘
```

### Рекомендации по развёртыванию

1. **PCI DSS**: для работы с данными банковских карт — использовать **GCP** (Workspace не сертифицирован по PCI DSS)
2. **Резидентность данных**: настроить регион хранения (US/EU) до начала обработки
3. **Минимальные привилегии**: Service Accounts с IAM-ролями только на чтение
4. **MCP Toolbox**: параметризованные SQL — LLM не получает прямой доступ к БД
5. **Аудит**: включить журналирование всех взаимодействий с Gemini
